프로젝트 소개

이 프로젝트는 절차적으로 작성된 스네이크 게임 구조를 객체지향 방식으로 재구성하는 것을 목표로 한다.
슬라이드에서 제시한 핵심은 뱀의 데이터를 전역 변수로 두지 않고 클래스 내부로 옮기고,
이동, 그리기, 먹이 처리, 충돌 처리, 연속 입력 처리를 각각 메서드와 상태로 정리하는 것이다.
현재 구현은 이 기본 구조를 반영하면서, 맵 확장, 여러 사과, 봇, 가속 게이지, 시각 효과 등을 추가한 확장판이다.

[STEP 1] 클래스 기본 틀
  > 슬라이드 목표
    - class Snake
    - __init__
    - self.bodies
    - 인스턴스 생성

  > 게임에서 보이는 기능
    - 플레이어 뱀과 봇 뱀이 각각 독립된 객체로 존재한다.
    - 각 뱀은 자신의 몸통, 방향, 생존 여부를 따로 관리한다.

  > 코드 위치
    - 84~100줄   : class Snake
    - 126~163줄 : reset()
    - 141~145줄 : 플레이어 객체 생성
    - 149~157줄 : 봇 객체 생성

  > 코드 역할
    - Snake 클래스는 뱀 1마리의 상태를 저장하는 기본 틀이다.
    - reset()에서는 플레이어와 봇을 각각 Snake(...) 형태로 생성한다.
    - 전역 SNAKE 대신 각 뱀 객체가 자기 몸 데이터를 직접 가진다.


[STEP 2] draw() 메서드
  > 슬라이드 목표
    - draw() 안에서 self.bodies를 순회하며 뱀을 그린다.
    - 그리기 로직을 묶어서 호출을 단순화한다.

  > 게임에서 보이는 기능
    - 뱀이 화면에 출력된다.
    - 머리, 몸통, 눈, 그림자가 함께 표시된다.
    - 사과와 UI도 함께 렌더링된다.

  > 코드 위치
    - 630~648줄 : draw_apples()
    - 649~722줄 : draw_snake()
    - 723~736줄 : draw_top_ui()
    - 737~764줄 : draw_bottom_ui()
    - 778~796줄 : draw()

  > 코드 역할
    - draw_snake()는 뱀 몸 좌표를 화면 좌표로 바꾸고 실제 외형을 그린다.
    - draw()는 배경, 사과, 뱀, UI를 순서대로 호출하여 한 프레임을 완성한다.
    - 슬라이드의 snake.draw() 개념을 확장한 부분이다.


[STEP 3] move() 메서드
  > 슬라이드 목표
    - 방향키 입력에 따라 머리를 이동시킨다.
    - 새 머리를 앞에 넣고 꼬리를 제거하여 전진을 만든다.

  > 게임에서 보이는 기능
    - 플레이어는 방향키 입력으로 이동한다.
    - 방향키를 누르고 있으면 연속 이동한다.
    - 봇도 일정 간격마다 다음 방향을 선택해 이동한다.

  > 코드 위치
    - 264~281줄 : handle_keydown()
    - 282~289줄 : handle_keyup()
    - 298~360줄 : update()
    - 507~607줄 : resolve_step()

  > 코드 역할
    - handle_keydown()은 방향 입력을 저장하고 첫 이동을 시작한다.
    - update()는 시간 기준으로 다음 이동 시점을 계산한다.
    - resolve_step()는 실제로 새 머리를 넣고, 먹지 않았으면 꼬리를 제거한다.
    - 슬라이드의 move(self, key) 기능이 입력 처리 + 이동 처리로 확장된 구조다.


[STEP 4] 먹이와 성장
  > 슬라이드 목표
    - 먹이를 먹으면 몸이 길어진다.
    - 먹힌 먹이는 제거되고 새 먹이가 생성된다.

  > 게임에서 보이는 기능
    - 사과를 먹으면 점수가 오른다.
    - 뱀 길이가 늘어난다.
    - 사과는 여러 개가 유지되며 계속 새로 생성된다.

  > 코드 위치
    - 160~163줄 : 시작 시 여러 사과 생성
    - 194~211줄 : spawn_apple()
    - 521줄     : 다음 칸이 사과인지 확인
    - 594~599줄 : 먹었을 때 성장 / 안 먹었을 때 꼬리 제거
    - 602~606줄 : 사과 보충

  > 코드 역할
    - spawn_apple()는 뱀 몸과 겹치지 않는 위치에 새 사과를 만든다.
    - 이동 결과가 사과 칸이면 pop()을 하지 않아 길이가 늘어난다.
    - 먹힌 사과는 삭제되고 개수가 유지되도록 새 사과가 추가된다.


[STEP 5] 충돌 감지와 게임오버
  > 슬라이드 목표
    - 벽 충돌, 몸 충돌을 감지한다.
    - 게임오버 여부를 판정한다.

  > 게임에서 보이는 기능
    - 벽에 머리가 닿으면 죽는다.
    - 몸통에 머리가 닿으면 죽는다.
    - 머리끼리 같은 칸에 충돌하면 둘 다 죽는다.
    - 플레이어가 죽으면 게임오버 화면이 뜬다.

  > 코드 위치
    - 507~607줄 : resolve_step()
    - 523~526줄 : 벽 충돌 판정
    - 554~556줄 : 몸통 충돌 판정
    - 558~565줄 : 같은 칸 머리 충돌
    - 568~575줄 : 머리 교차 충돌
    - 579~581줄 : 죽은 뱀 처리
    - 352~355줄 : 플레이어 사망 시 게임오버 전환
    - 765~777줄 : 게임오버 화면 출력

  > 코드 역할
    - 다음 이동 위치를 기준으로 충돌 여부를 먼저 확인한다.
    - 충돌한 뱀은 alive = False 처리된다.
    - 플레이어가 죽으면 전체 게임을 종료 상태로 바꾸고 GAME OVER 화면을 보여준다.
    - 슬라이드의 return is_game_over 개념을 다중 뱀 구조에 맞게 상태 갱신 방식으로 확장한 부분이다.


[STEP 6] 방향키 홀드 연속 이동
  > 슬라이드 목표
    - 방향키를 누르고 있으면 반복 입력이 발생하게 하여 더 빠르게 이동하도록 만든다.

  > 게임에서 보이는 기능
    - 방향키를 짧게 누르면 1회 이동한다.
    - 방향키를 계속 누르면 연속 이동한다.
    - 가속 게이지와 결합되어 더 빠른 이동 구조로 확장되어 있다.

  > 코드 위치
    - 264~281줄 : 방향키 입력 시작
    - 282~289줄 : 방향키 입력 해제
    - 303~306줄 : 홀드 시간 누적
    - 323~327줄 : 현재 이동 간격 계산
    - 329~332줄 : 실제 연속 이동 실행
    - 334~336줄 : 가속 에너지 감소

  > 코드 역할
    - held_dir는 현재 누르고 있는 방향을 저장한다.
    - hold_time, move_timer는 얼마나 오래 누르고 있었는지와 다음 이동 시점을 계산한다.
    - 슬라이드에서는 pygame.key.set_repeat()를 사용하지만, 현재 구현은 같은 기능을 직접 타이머 방식으로 만든 구조다.
